### 1.2 프로젝트 목적

이 프로젝트는 **목적이 두 개**이며, 그 우선순위가 명확히 정해져 있다. 이 우선순위를 모르면 모든 의사결정을 오독하게 된다.

| 순위 | 목적 | 설명 |
|------|------|------|
| **1순위** | **이직 포트폴리오 / 제품 빌딩 역량 증명** | 마일로는 현직 서비스 기획자(SOOP 플랫폼 BM팀)이며, 대형 IT사 서비스 기획 직군으로의 이직을 목표로 한다. 이 프로젝트는 "문제 정의 → IA 설계 → 직접 구현 → 출시 → 트래픽/수익 데이터 분석까지 **혼자 전 사이클 수행**"이라는 면접 서사를 실물로 증명하기 위한 것이다. |
| **2순위** | **수익화 (AdSense → 구독)** | 실제 수익 파이프라인을 구축하고 운영해본다. 다만 **AdSense 수익 자체는 이직 시점까지 유의미한 금액이 되지 않는다는 것을 이미 계산으로 확인했고, 마일로도 이를 수용했다.** 수익은 "포트폴리오의 증거(월 OO 트래픽, 수익 OO원)"로서 더 큰 가치를 가진다. |

**이 우선순위가 실제 의사결정에 미친 영향의 예:**

- Notion + Super.so 조합을 **거부**한 이유 중 하나는 "커리어 포트폴리오 서사가 약함 — '콘텐츠 운영'으로만 읽힘"이었다. 기술적 편의보다 서사를 택했다.
- Phase 2에서 Supabase + Stripe를 도입하기로 한 이유 중 하나는, 결제·구독 시스템을 **직접 구현해보는 학습 경험** 자체가 목적이기 때문이다. 순수 수익 관점만이라면 훨씬 단순한 선택지가 있었다.
- 정부 창업 지원금은 "트래픽 지표 확보 및 구독 모델 전환 이후"로 미뤘다. 지금 단계(정적 사이트 + 광고 모델 + 재직 중)는 지원 프로그램 요건과 맞지 않는다고 판단.

> **AI에게**: 어떤 제안을 할 때 "이게 수익에 더 좋다"만으로는 부족하다. "이게 포트폴리오 서사에 어떤 영향을 주는가"를 함께 답해야 마일로의 의사결정 기준에 맞는다.

### 1.3 서비스 소개

**GetPassLab은 산업안전기사 수험생을 위한 "기출 역설계 기반 핵심 요약 서비스"다.**

한 문장으로 요약하면:
> 5개년 기출문제 1,675문항을 전수 DB화하고, 그 데이터를 역으로 분석해 "실제로 출제되는 것만" 챕터 단위로 정리해 보여주는 정적 웹사이트.

핵심 콘텐츠 단위는 **챕터(chapter)** 다. 챕터 1개 = 1개의 독립된 학습 주제 = 1개의 URL = 1개의 SEO 랜딩 페이지.

각 챕터는 5개 섹션으로 구성된다:

| 섹션 | 내용 | 생성 방식 |
|------|------|-----------|
| 1. 핵심 공식 / 정의 | 공식, 정의, 함정 주의 인용블록 | **수동 작성** |
| 2. 단위·기호 해설 | 기호/명칭/단위/주의점 표 | **수동 작성** |
| 3. 해석 / 의미 | 왜 이 공식이 중요한지, 어디에 응용되는지 | **수동 작성** |
| 4. 기출 출제 이력 | 출제 빈도 라벨 + 회차 뱃지 + (클릭 시) 문제 레이어 | **빌드 타임 자동 생성** (수동은 "출제 경향 코멘트" 1~2줄뿐) |
| 5. 관련 챕터 | 동일 그룹 + 수동 지정 챕터 링크 | **빌드 타임 자동 생성** |

> **섹션 4가 이 서비스의 차별화 핵심이다.** 이건 대화 중 실제로 검증된 판단이다. 옴의 법칙 샘플 챕터를 실데이터로 채워보니 *"옴의 법칙은 단독 출제 0건, 응용 결합형 100%"* 같은, **시중 수험서에 없는 인사이트**가 데이터에서 자동으로 도출됐다. 기출 DB를 전수 보유한 서비스만이 쓸 수 있는 콘텐츠이며, 이것이 GetPassLab의 방어 가능한 해자(moat)다.

### 1.4 해결하려는 문제

**시장 관찰 (프로젝트 착수 근거)**

| 관찰 | 내용 |
|------|------|
| 기존 사이트의 편중 | comcbt 등 기존 산업안전기사 사이트는 **문제풀이(CBT) 특화**다. 문제를 풀 곳은 많다. |
| 빈 공간 | **구조화된 "요약·정리" 사이트가 없다.** 수험생은 문제를 풀다 막히면 개념을 찾아야 하는데, 그때 갈 곳이 블로그 단편 글이나 유료 수험서뿐이다. |
| 수험생의 실제 페인포인트 | "이 챕터가 실제로 시험에 나오긴 하는가?", "어떤 형태로 나오는가?"에 대한 답이 없다. 수험서는 전 범위를 균등하게 다루므로 **출제 빈도 정보가 소실**되어 있다. |
| 검색 수요 | 산업안전기사는 연 3회 시험 → **회차마다 트래픽 스파이크**가 반복 발생. 검색 수요가 주기적으로 확보되는 구조. |

**그래서 GetPassLab이 해결하는 문제는 정확히 이것이다:**

1. **"무엇을 공부해야 하는가"의 부재** → 기출 역설계로 출제 빈도를 명시한다.
2. **"이 개념이 어떻게 출제되는가"의 부재** → 챕터마다 실제 기출 회차 뱃지를 붙이고, 클릭하면 실제 문제를 보여준다.
3. **"핵심만 빠르게 훑을 곳"의 부재** → 챕터당 700~1,300자의 명사형 요약. 5~10분 완결.

**마일로가 이 문제를 정확히 아는 이유 (중요)**

마일로는 **본인이 산업안전기사 필기에 합격했고, 현재 실기를 준비 중**이다. 즉 타겟 사용자의 페인포인트를 1인칭으로 겪었다. 이는 두 가지 실질적 이점을 준다:
- 콘텐츠를 직접 생산·검수할 수 있다 (외주·AI 전량 의존이 아님).
- 실기 콘텐츠는 본인의 학습 진도와 콘텐츠 생산이 동시에 진행된다.

> **참고 (초기 자기검증)**: 마일로는 프로젝트 초기에 "나는 산업안전 도메인 전문가가 아닌데, AI 큐레이션에 의존하는 콘텐츠가 괜찮은가?"라는 의문을 스스로 제기했다. 이에 대한 결론은 **"산업안전기사는 한국산업인력공단이라는 공인 원본(official source)이 존재하고 정답이 명확하며 갱신 주기가 예측 가능하다"** 는 것이었다. 정답이 없는 주제(예: 폐기된 '조합 추천 사이트' 아이디어)와 근본적으로 다른 구조이며, 이것이 이 주제를 선택한 결정적 근거 중 하나다.

### 1.5 프로젝트 비전

**단기(~2026년 상반기)**: 산업안전기사 필기 영역에서 "검색하면 나오는 요약 사이트"의 지위를 확보한다. AdSense 승인 → 시험 시즌 트래픽 흡수 → 실측 데이터 확보.

**중기(Phase 2)**: 정적 사이트를 웹앱으로 전환한다. 로그인·학습 진도·오답 노트를 붙여 **일회성 검색 유입을 재방문 사용자로 전환**한다. 이 시점에서 서비스는 "콘텐츠 사이트"에서 "학습 도구"로 성격이 바뀐다.

**장기(Phase 3 이후)**: 자격증을 순차 확장하여 **자격증 학습 플랫폼**으로 포지셔닝한다. 확장 순서는 이미 확정되어 있다(§5 참조). 최종적으로 "EdTech 개인화 학습 플랫폼"의 서사를 갖추면, 정부 지원금·투자·매각 등 다양한 출구가 열린다.

> **매각 옵션도 검토된 바 있다.** Flippa 등 사이트 거래 플랫폼에서 웹사이트 가치평가는 통상 **월 순이익 × 20~40배**로 형성된다는 점을 확인했다. 현재는 실행 계획이 아니라 "출구 옵션 중 하나로 인지하고 있는 상태"다.

## 2. 타겟 사용자

### 2.1 1차 타겟 (현재 서비스 대상)

**산업안전기사 필기 응시생**

### 2.2 2차 타겟 (확장 예정)

**산업안전기사 실기 응시생**

- 블루오션. 실기 요약 콘텐츠는 필기보다 훨씬 부족하다.
- 마일로 본인이 실기 준비 중 → 학습과 콘텐츠 생산이 동기화됨.
- 다만 **트래픽 규모는 필기 대비 작다.** 실기 시즌(4월경)까지 점진 보충하는 방식.
- MVP에서는 **제외**했다. 근거: 1월 필기 시즌 트래픽을 잡는 것이 우선순위상 압도적으로 크다.

### 3.2 설계됐으나 구현 상태가 불확실한 기능

**과목 탭 바 (Subject Tab Bar)**

- **배경**: 수험생은 한 학습 세션에 여러 과목을 오간다. 그런데 현재 구조에서 과목 이동은 GNB 드롭다운 또는 브레드크럼 역행뿐이라 2~3클릭이 든다. 모바일에서는 챕터 사이드바가 숨어 있어 사실상 경로가 하나뿐이다.
- **목표**: 필기 하위 어느 페이지에서든 **과목 이동이 1클릭**.
- **사양**: 6과목 가로 탭("1과목"~"6과목" 짧은 라벨). 현재 과목 활성 하이라이트. 배치는 과목 목차·챕터 상세 페이지의 브레드크럼 아래 공통 삽입. 모바일은 가로 스크롤 탭 + 현재 과목이 처음부터 뷰포트 안에 보이도록.
- **진행**: Claude Design에 전달할 지시문까지 작성 완료.
- ⚠️ **확실하지 않음**: 실제 구현·배포 완료 여부.

### 3.3 광고 배치 (확정 설계)

| 페이지 | 광고 수 | 위치 |
|--------|---------|------|
| 홈 | 1 | 6과목 카드 그리드와 인기 챕터 사이 |
| 자격증 허브 | 0 | — |
| 필기 (시험구분) | 1 | 6과목 카드와 인기 챕터 사이 |
| 과목 페이지 | 2 | 그룹 1·2 사이 + 페이지 하단 |
| **챕터 페이지** | **3** | **섹션 3·5 사이 + 페이지 하단 (수익 핵심)** |
| 검색 모달 | 0 | — |
| /terms, /privacy | 0 | — |

> **수익의 80% 이상이 챕터 상세 페이지에서 발생할 것으로 예상**한다. 이것이 챕터 페이지를 SEO 랜딩 페이지로 설계한 이유이며, 챕터를 잘게 쪼갠 이유이기도 하다.

**AdSense 포맷 전략**: 배너 단독보다 **Rewarded Interstitial 병행**이 RPM 향상에 유리하다고 판단. 경쟁 CBT 사이트가 "광고 시청 후 6시간 이용 게이트" 구조를 쓰는 것을 실제로 조사해 확인했고, 그것이 AdSense Rewarded Interstitial 포맷임을 확인했다. (※ 이 조사 과정에서 Claude가 "AdSense로는 불가능하다"고 잘못 답했다가 마일로가 제시한 증거로 정정한 이력이 있다. — 이 사례는 AI가 광고 정책을 단정하면 안 된다는 교훈으로 기록한다.)

## 6. 서비스 철학

> 이 섹션은 "GetPassLab이 사용자에게 어떤 태도를 취하는가"를 정의한다. 새 기능을 제안할 때 이 철학과 충돌하지 않는지 검증해야 한다.

### 6.1 데이터가 주장을 대신한다

수험 콘텐츠는 "이게 중요합니다"라는 저자의 주관으로 가득하다. GetPassLab은 그 자리에 **숫자**를 놓는다. "5개년 중 4회 출제", "단독 출제 0건, 응용 결합형 100%" 같은 진술은 **기출 DB에서 자동 도출**된 것이지 누가 주장한 것이 아니다.

이것이 서비스의 신뢰 기반이자, 동시에 **경쟁 사이트가 복제하기 어려운 지점**이다. 기출 전수 DB를 갖고 있지 않으면 이 문장을 쓸 수 없다.

**따라서 금지되는 것**: 근사·뭉뚱그리기. 실제로 "2018~2022 공통 출제"라는 표기를 썼다가 폐기한 이력이 있다. 사유: *"데이터 서비스에서 이런 근사 표기는 신뢰도를 깎는다."* **개별 회차를 명시하거나, 아예 쓰지 않는다.**

### 6.2 억지로 늘리지 않는다 (자연 분량 원칙)

SEO를 위해 분량을 부풀리는 것을 금지한다.

- 초기에는 "고빈도 챕터 최소 1,500자"라는 기준이 있었다. 그러나 이는 **통합형(굵은) 챕터**를 전제한 수치였다.
- 기출 기반으로 챕터를 잘게 재편성한 뒤, 단일 챕터의 **자연 분량은 1,000~1,300자**로 재산정됐다.
- v3 템플릿 확정 후 수작업 분량은 **섹션 1~3 본문 + 코멘트 1줄 ≈ 700~900자/챕터**로 더 줄었다.
- **원칙: 억지로 늘리면 품질이 떨어진다. 분량 기준은 하한선이 아니라 관찰값이다.**

### 6.3 무료가 기본이다

핵심 요약 콘텐츠는 로그인 없이 무료로 접근한다. 이는 광고 모델의 요구이기도 하지만, 그 이상으로 **수험생의 사용 맥락**(문제 풀다 막혀서 급히 검색)에 로그인 게이트가 치명적이기 때문이다.

이 원칙 때문에 **의도적으로 수용한 보안 트레이드오프**가 있다: 정적 사이트에서 `questions.json`을 클라이언트에 내려주므로 **정답 데이터가 노출된다.** 이는 무료 시험 준비 서비스의 업계 표준이며, "정답을 숨기기 위해 서버를 도입"하는 것은 비용 대비 무의미하다고 판단했다.

### 6.4 익명 운영 (MVP 한정)

MVP 단계에서 푸터에 운영자 정보를 노출하지 않는다. 서비스가 개인의 사이드 프로젝트가 아니라 **하나의 제품**으로 보이는 것이 사용자 신뢰에 유리하다는 판단.

> ⚠️ **확실하지 않음**: AdSense 승인 요건 및 개인정보처리방침 작성 시 사업자·운영자 정보 표기가 필요할 수 있다. 이 원칙과 충돌 가능성 있음. 신청 전 재검토 필요.

## 7. 개발 철학

> 이 섹션은 "코드와 데이터를 어떻게 다루는가"를 정의한다. **여기 적힌 원칙을 어기는 제안은 하지 말 것.**

### 7.1 동결 키 (Frozen Keys) — 최우선 원칙

**절대 변경 불가한 식별자 3종:**

| 키 | 역할 |
|----|------|
| `slug` | ① 챕터 URL 식별자 ② 기출↔챕터 조인 키 ③ (Phase 2) 유저 학습 기록의 참조 대상 |
| `subject_id` | ① 과목 URL 세그먼트 ② 조인 키 |
| `question_id` | ① 기출문제 고유 식별자 ② (Phase 2) 오답 노트의 참조 대상 |

**왜 동결인가**: 이 세 필드는 **URL 식별자 · 조인 키 · 유저 기록 참조**를 동시에 담당한다. 변경하면 ① 검색 인덱스가 깨지고(SEO 손실) ② 기출-챕터 관계가 끊기고 ③ Phase 2의 유저 데이터가 고아(orphan)가 된다.

**실무 규칙**
- 슬러그는 CSV/기존 값을 **그대로** 쓴다. Cowork나 AI가 **임의 생성·수정 절대 금지**.
- 슬러그 공란 행은 **생성하지 말고 보고**한다. (잘못 만든 슬러그를 나중에 고칠 수 없기 때문)

**과목 슬러그 (동결 완료)**

| 과목 | 한글명 | 슬러그 | `subject_id` |
|------|--------|--------|--------------|
| 1과목 | 산업재해 예방 및 안전보건교육 | `safety-management` | 1 |
| 2과목 | 인간공학 및 위험성평가·관리 | `ergonomics` | 2 |
| 3과목 | 기계·기구 및 설비 안전관리 | `mechanical` | 3 |
| 4과목 | 전기설비 안전관리 | `electrical` | 4 |
| 5과목 | 화학설비 안전관리 | `chemical` | 5 |
| 6과목 | 건설공사 안전관리 | `construction` | 6 |

> **폐기된 구(舊) 과목 슬러그** (문서·코드에서 발견되면 **레거시**이므로 갱신 대상):
> `accident-prevention` / `human-engineering` / `machine-safety` / `electrical-safety` / `chemical-safety` / `construction-safety`
> 이는 URL 구조를 `/subjects/{과목}/{챕터}`에서 `/industrial-safety/written/{과목}/{챕터}/`로 재편하면서 함께 교체됐다.

**문제 ID 형식**: `{연도}{시험일}_{문제번호 3자리}` — 예: `20220424_061`, `20200926_073`

### 7.2 단일 진실 원천 (이중 원본 절대 금지)

이 원칙은 **실제 고통을 겪고 얻은 것**이다.

**겪은 문제**: 노션에서 챕터 메타데이터(작성 상태 등)를 관리하고, 본문은 md 파일로 관리했다. Cowork가 챕터를 쓰면 → 노션에 가서 작성 상태를 **또** 갱신해야 했다(이중 기입). 동기화가 어긋나면 **장부(노션)와 실물(md)이 다른 상태**가 된다. 이 불일치가 반복 발생했다.

**해결**: 노션을 **동결(읽기 전용 아카이브)** 하고, 진실 원천을 git으로 단일화했다.

- 노션 표 = "사람이 갱신하는 장부"
- `/admin/` 페이지 = "실물(md 파일)에서 자동 생성되는 장부" → **어긋날 방법이 물리적으로 없다**

**최종 데이터 아키텍처 (확정)**

| 데이터 | 저장소 | 관리 방식 |
|--------|--------|-----------|
| 챕터 본문 | md 파일 (git) | 개별 수정 = 직접/Cowork, 일괄 수정 = Cowork |
| 챕터 메타 | md **frontmatter** (git) | `/admin/` 표로 조회 |
| 기출문제 | `questions.json` (git) | Cowork |
| 기출↔챕터 관계 | **챕터 frontmatter의 `questions` 배열** — **유일한 저장처** | Cowork |
| 문서류 (정책서 등) | 노션 (무료 플랜) | 읽기 전용 |
| 사용자 데이터 (Phase 2) | Supabase | 그때 설계 |

**본문을 DB화하지 않기로 한 이유** (명시적으로 검토 후 기각):

| 기준 | 메타데이터 | 본문 |
|------|-----------|------|
| 데이터 형태 | 짧은 정형 값 → 표 셀에 적합 | LaTeX 수식·표·인용 혼재 장문 → **셀에 넣는 순간 관리 불능** |
| 조회 니즈 | "미작성 몇 개?" 같은 집계 필요 | 본문을 집계할 일이 없음. 읽기는 사이트에서 |
| 수정 패턴 | 일괄 (74건 상태 변경) | 개별 (한 줄 수정) |
| 이력 추적 | 최신값만 중요 | **git이 줄 단위 변경 이력 제공 — 본문 관리의 핵심 인프라** |

> 본문을 DB에 넣으면 "노션 → md 변환·동기화" 계층이 **부활**한다. 우리가 없앤 바로 그 계층이다.

### 7.3 콘텐츠는 역설계로 만든다

**확정된 순서 (역행 금지)**

```
① 기출문제 DB 전수 구축
      ↓
② 기출 기준으로 챕터 체계 재정비 (출제되지 않는 챕터는 후순위/삭제, 새 챕터 도출)
      ↓
③ 기출을 기준으로 본문 작성 (실제 출제된 각도로만 서술)
```

**왜 이 순서인가**: 챕터를 먼저 만들고 기출을 나중에 붙이면, "출제되지 않는 내용을 열심히 쓰는" 낭비가 발생한다. 기출을 먼저 알면 **무엇을 쓰지 않아도 되는지**를 알 수 있고, 이것이 1인 운영에서 가장 큰 레버리지다.

### 7.4 웹앱 확장을 전제로 초반부터 설계한다

정적 사이트지만, Phase 2(로그인·진도·오답)를 붙일 수 있는 구조로 처음부터 짓는다.

- 데이터 모델을 **콘텐츠 레이어 / 유저 레이어 2층**으로 분리 설계했다. 유저 레이어(users, attempts, wrong_notes, progress, bookmarks)는 지금 구현하지 않지만 **스키마는 이미 그려져 있다.**
- 동결 키 원칙이 그 접착제다. 지금 정한 `slug`가 나중에 유저 학습 기록의 외래 키가 된다.

### 7.5 MVP 규율 — 과도한 복잡도 회피

- "1월 15일에 게시 가능한 최소 단위"로 계속 좁혔다. (※ 이 날짜는 §4.4의 불확실성 참조)
- 판단 기준: **"기능 부족은 트래픽 손실이 적지만, 콘텐츠 부족은 치명적이다."** AdSense 승인 기준도 기능 다양성이 아니라 **콘텐츠 충실도**다.
- 콘텐츠 제작이 유일한 진짜 병목이다 (1인 + 학업 병행).

|------|------|
| **마일로** | 기획·의사결정·콘텐츠 방향성·검수. **코드는 직접 쓰지 않는다** |
| **메인 대화 (Claude)** | 고수준 설계, 의사결정 근거 제시, **지시문 초안 작성**, 아키텍처 판단 |
| **Cowork** | 반복·자동화 실행 (PDF 파싱, 파일 일괄 생성·수정, 배포). 메인 대화에서 만든 지시문을 받아 실행 |

**Cowork 운영 규칙 (확정)**
- 세션은 **과목 단위로 분할**한다 (한 번에 254개를 시키지 않는다)
- **시작 전 Pull, 완료 후 Push**
- **동시 세션 금지** (충돌 방지)

### 7.7 의사결정 프로토콜

> **AI는 이 프로토콜을 따를 것.**
>
> 1. **결론을 먼저 말한다.** 서론·아부·이모티콘 금지.
> 2. **선택지를 표로 제시하고 각각의 장단을 명시한다.**
> 3. **추천안과 그 근거를 명시한다.**
> 4. **마일로가 검토 후 결정한다.** AI가 대신 확정하지 않는다.
> 5. 불확실한 정보는 **반드시 불확실하다고 말한다.** 수치·법령·기준은 최신 여부를 검토한 뒤 답한다.
> 6. 코드 구현 단계의 상세 설명은 불필요하다. **"어디에 붙여넣는지"만 명확히** 안내하고 완성본 코드를 준다.

## 8. 지금까지 프로젝트에서 가장 중요했던 의사결정

> 시간순이 아니라 **영향력 순**으로 배열했다. 각 항목은 [결정 / 대안 / 근거 / 파급효과] 구조다.

### D-1. 주제 선정 — 산업안전기사 요약 서비스

- **결정**: 산업안전기사 수험생 대상 핵심 요약 웹서비스
- **탐색했던 대안 6~7종**: ① 자격증 CBT 사이트 ② 공공데이터 시각화 서비스 ③ 업종별 법령/규정 요약 사이트 ④ 채용 JD 분석/비교 툴 ⑤ 연봉/처우 계산기 모음 ⑥ 소규모 사업자 세금/행정 가이드 ⑦ (별도 검토) 일상 조합 추천 사이트
- **근거**:
  - 마일로가 **직접 필기 합격 → 실기 준비 중** → 페인포인트를 1인칭으로 앎
  - 기존 사이트(comcbt 등)는 **문제풀이 특화** → 구조화된 요약 사이트가 **빈 공간**
  - **연 3회 시험** → 트래픽 스파이크가 주기적으로 반복
  - **공인 원본(한국산업인력공단)이 존재** → 정답이 명확하고 갱신 주기 예측 가능. ("조합 추천 사이트"를 기각한 결정적 차이가 이것이다 — 그쪽은 canonical source가 없어 주관성·유지보수 부담이 큼)
- **파급효과**: 이후 모든 콘텐츠 전략·확장 전략이 "기출 데이터가 존재하는 자격증"이라는 전제 위에 세워졌다.

### D-2. Notion + Super.so 조합 기각 → 직접 구현

- **결정**: 노코드(Notion + Super.so) 대신 **Astro로 직접 구현**
- **근거 (3가지, 모두 중요)**:
  1. **AdSense 삽입 제약** — 수익 모델이 광고인데 광고를 못 넣으면 근본 모순
  2. **커리어 포트폴리오 서사 약화** — "콘텐츠 운영"으로만 읽힘. 1순위 목적(§1.2)과 정면 충돌
  3. **웹앱 확장 시 구조적 한계** — Phase 2로 갈 길이 막힘
- **파급효과**: 이 결정이 없었다면 "혼자 전 사이클 수행"이라는 면접 서사 자체가 성립하지 않는다. **프로젝트의 정체성을 규정한 결정.**

### D-3. 기출 역설계 — 콘텐츠 전략의 근본

- **결정**: 기출 DB 먼저 → 챕터 재정비 → 본문 작성
- **대안**: 수험서 목차대로 챕터를 만들고 콘텐츠를 채운 뒤, 나중에 기출을 태깅
- **근거**: 무엇을 **안 써도 되는지**를 먼저 아는 것이 1인 운영의 최대 레버리지. 그리고 이것이 §6.1의 차별화("데이터가 주장을 대신한다")를 가능케 하는 유일한 순서.
- **파급효과**: 기출 DB 구축(1,675문항)이 콘텐츠 작성보다 **선행 작업**이 됐고, 그 때문에 프로젝트 초중반의 대부분이 데이터 파이프라인 작업이었다.

### D-4. 동결 키 3종 지정 (`slug`, `subject_id`, `question_id`)

- **결정**: 이 3개 필드는 한 번 확정 후 **절대 변경 불가**
- **근거**: URL 식별자 + 조인 키 + (미래) 유저 기록 참조를 동시에 담당하므로, 변경 시 SEO·데이터 무결성·유저 데이터가 동시에 붕괴
- **파급효과**: 이후 모든 자동화 지시문에 "슬러그 임의 생성 금지", "공란은 보고만" 규칙이 들어갔다. **이 프로젝트에서 가장 자주 인용되는 제약.**

### D-5. 챕터 섹션 구조 7섹션 → 5섹션 축소, 그리고 v3 템플릿

- **결정**: 챕터 = 5섹션. 그중 **섹션 4·5는 빌드 타임 자동 생성**
- **대안 (폐기)**: 7섹션 구조 / 섹션 4의 기출 이력을 수동 표로 작성
- **근거**:
  - 수동 작성 영역을 줄여야 254개 양산이 가능하다
  - 기출 이력은 frontmatter의 `questions` 배열 + `questions.json` 조인으로 **빌드 타임 자동 생성 가능**
  - 본문에 링크를 하드코딩하면 254개 유지보수가 **불가능**해진다
- **파급효과**: 챕터당 수작업이 **700~900자**로 확정. 원래 추정의 **절반 이하**. 이 결정 덕에 74개 양산이 실제로 완료될 수 있었다.
- **부수 결정**: 문체를 **명사형 어미**로 전면 통일 (수험 요약의 표준 문체 + 스캔 가독성)

### D-6. 노션 동결 → git 단일화, `/admin/` 페이지 도입

- **결정**: 노션은 **읽기 전용 아카이브**. 챕터 메타는 md frontmatter로 흡수. 현황 조회는 빌드 타임 생성되는 `/admin/` 페이지가 담당
- **대안 (폐기)**: 노션을 계속 작업 도구로 유지 / 본문까지 DB화
- **직접적 계기**: ① 노션 플랜 제한으로 MCP 행 조회가 막힘 ② **이중 기입 → 장부-실물 불일치**가 반복 발생
- **근거**: "실물에서 자동 생성되는 장부는 어긋날 방법이 없다"
- **인지된 한계 (수용함)**: `/admin/`은 **조회 전용**. 노션처럼 셀 클릭 수정 불가. 수정은 Cowork 지시 또는 GitHub 웹 편집. → 마일로의 워크플로가 이미 "편집은 지시, 확인만 직접"이므로 실질 손실 없다고 판단.
- **부수 결정**: 미작성 챕터(1차+2차)를 **frontmatter만 있는 스텁 md**로 전량 생성. 254개 전체가 저장소에 존재하되 `status: 미작성`은 빌드에서 제외. → 이후 양산 작업이 "스텁 채우기"가 되어 **노션 의존이 완전히 소멸**.
- **또 하나의 합의**: *"완벽하게 정리 후 이관"이 아니라 "이관 후 다듬기"* — 어차피 이관하면 다듬는 작업이 git에서 훨씬 쉬워진다.

### D-7. URL 구조 재편 — 자격증 허브 계층 추가

- **결정**: `/subjects/{과목}/{챕터}` → **`/industrial-safety/written/{과목}/{챕터}/`**
- **근거**: 자격증 확장(§5.3)이 확정된 이상, URL에 **자격증 계층과 시험구분(필기/실기) 계층**이 없으면 확장 시 전면 재편이 강제된다. 그때는 이미 SEO 인덱스가 쌓여 있어 이동 비용이 크다.
- **파급효과**: 과목 슬러그 6종이 이 시점에 함께 교체되고 **동결**됐다.
- **교훈**: **동결 키는 "동결하기 직전"이 마지막 수정 기회다.** 확장 계획이 확정된 뒤에 URL을 동결한 것은 옳은 순서였다.

### D-8. 브랜드 = 도메인 = GetPassLab

- **결정**: 브랜드명을 도메인과 완전 일치시킴 (§1.1 상세)
- **근거**: "Get(합격을 얻는다)"의 행동 유도형 의미 + 도메인·브랜드 일치의 SEO/기억성 이점
- **비고**: Claude의 권장안(A: 브랜드는 PassLab)과 다른 선택. **마일로의 의미론적 판단이 관례를 이겼다.**

### D-9. 확장 방향 — 무관 카테고리 금지, 인접 자격증만

- **결정**: 산업안전기사 → **컴퓨터활용능력 2급** → 공조냉동기계산업기사 등
- **폐기안**: 선거·정부 정책 등 무관 카테고리 추가로 트래픽 확대
- **근거**: 도메인 정체성 파편화 / SEO 페널티 / 광고주 기피 / **AdSense 계정 자체가 위험**
- **재조정 이력**: 2순위는 원래 공조냉동기계산업기사였으나, **컴활 2급의 응시자 규모(연 약 50만)** 가 압도적이라 승격됨

### D-10. Phase 2 스택 — Supabase + Stripe

- **결정**: 인증 = Supabase Auth, 결제 = Stripe Checkout + Webhook
- **근거**: 공식 통합 지원 / 반복 결제 지원 / **B2C 업계 표준이라 포트폴리오 가치가 높음**
- **숨은 목적 (명시적으로 언급됨)**: 마일로는 기능 매트릭스를 그리려는 게 아니라 **결제·구독 시스템을 실제로 만들어보는 것 자체가 목표**라고 밝혔다. 이 발언이 스택 선택을 좌우했다.

### D-11. 정적 사이트의 보안 트레이드오프 수용

- **결정**: `questions.json`을 클라이언트에 내려줌 → **정답 노출을 감수**
- **근거**: 무료 시험 준비 서비스의 업계 표준. 정답을 숨기려고 서버를 도입하는 것은 비용 대비 무의미
- **파급효과**: 문제 레이어 기능을 정적 사이트에서 구현할 수 있게 됨

### D-12. Mermaid를 ERD 주력 도구로

- **결정**: ERD는 **Mermaid** 텍스트로 관리. dbdiagram.io는 Phase 2에서 SQL export가 필요해질 때 보완재로
- **근거**: 스키마가 계속 바뀌는 단계에서는 **텍스트 편집이 압도적으로 빠름**. 노션·GitHub에서 네이티브 렌더링되어 이미지 재업로드가 불필요. **"설계를 코드로 관리한다"는 포트폴리오 서사**와도 맞음

## 9. AI가 반드시 이해해야 하는 프로젝트 컨텍스트

> **이 섹션은 이 문서에서 가장 중요하다.** 새로 투입되는 AI가 사고 없이 작업하기 위한 최소 조건이다.

### 9.1 절대 하지 말아야 할 것 (Hard Constraints)

| # | 금지 사항 | 이유 |
|---|-----------|------|
| 1 | **`slug` / `subject_id` / `question_id` 를 변경·재생성한다** | 동결 키. URL·조인·유저 데이터가 동시에 붕괴 |
| 2 | **슬러그를 임의로 만들어낸다** | 위와 동일. 슬러그가 없으면 **만들지 말고 보고**한다 |
| 3 | **같은 정보를 두 곳에 저장한다 (이중 원본)** | 이 프로젝트가 가장 크게 데인 문제 |
| 4 | **노션을 수정한다** | 노션은 **동결된 읽기 전용 아카이브**다 |
| 5 | **챕터 본문에 기출 링크를 하드코딩한다** | 254개 유지보수 불가능. 기출은 frontmatter `questions` 배열 → 빌드 타임 조인 |
| 6 | **SEO를 위해 본문 분량을 부풀린다** | 자연 분량 원칙 위반 |
| 7 | **무관한 카테고리(정치·선거 등)를 콘텐츠에 추가한다** | AdSense 계정 위험 |
| 8 | **"2018~2022 공통" 식의 근사·뭉뚱그린 데이터 표기를 쓴다** | 데이터 서비스의 신뢰도를 깎는다. 개별 회차를 명시하거나 쓰지 않는다 |
| 9 | **Cowork 세션을 동시에 여러 개 돌린다** | 파일 충돌 |
| 10 | **마일로 대신 최종 결정을 내린다** | 마일로는 **근거를 받고 본인이 결정**한다. AI는 선택지와 추천안까지만 |

### 9.2 반드시 참조해야 할 진실 원천 (Source of Truth)

| 알고 싶은 것 | 봐야 할 곳 |
|--------------|-----------|
| 챕터가 몇 개인지 / 작성 상태 | **`/admin/` 페이지** 또는 `src/content/chapters/` 실제 파일 |
| 챕터 메타데이터 | 각 md의 **frontmatter** |
| 기출문제 데이터 | **`questions.json`** |
| 기출↔챕터 관계 | 챕터 frontmatter의 **`questions` 배열** (유일한 저장처) |
| 스키마 정의 | **`src/content.config.ts`** (Zod 스키마) |
| 과거 정책 문서 | 노션 (읽기 전용. **최신이 아닐 수 있음**) |

> **이 문서(PROJECT_CONTEXT.md)조차 진실 원천이 아니다.** 코드와 데이터가 진실이다. 이 문서는 "왜 그렇게 됐는가"를 설명할 뿐이다.

### 9.4 도메인 지식 — 산업안전기사 기본

새 AI가 콘텐츠를 다루려면 이 정도는 알아야 한다.

| 항목 | 값 |
|------|-----|
| 자격증 | 산업안전기사 (한국산업인력공단) |
| 시험 구성 | 필기 + 실기 |
| 필기 과목 | **6과목** |
| 필기 문항 | **과목당 20문제 × 6과목 = 120문제** |
| 시험 횟수 | **연 3회** |
| 기출 DB 보유 범위 | **5개년 14회차** (2018~2022 기준으로 언급됨), **1,675문항** (중복·정답 공란 제외) |

> ✅ **문항 수 계보 (해소됨)** — `ARCHITECTURE.md` §7.2, `DECISION_LOG.md` D-56에서 변환 코드로 확정.
> 노션 CSV 약 **1,690**행(원본) → 중복 10건 제거 → **1,680** → 정답 파싱 실패 5건 제외 → **1,675** (`questions.json` 최종 탑재값).
> 세 숫자 모두 맞고 단계가 다를 뿐이다. **최종 탑재값 = 1,675.** (실측: `python3 -c "import json;print(len(json.load(open('src/data/questions.json'))))"`)
>
> ⚠️ **확실하지 않음 — 기출 연도 범위**
> "2018~2022년 5개년"과 "7개년"(대화 제목: *코워크 7개년 기출 DB화*) 표현이 혼재한다. 최종 DB에 실제로 포함된 회차 목록은 `questions.json`에서 직접 확인할 것.

### 9.5 마일로에 대해 알아야 할 것

| 항목 | 내용 |
|------|------|
| 직무 | IT 서비스 기획자 (현직: SOOP 플랫폼 BM팀) |
| 목표 | 대형 IT사 서비스 기획 직군으로 이직 |
| 기술 이해도 | **기술 개념은 이해한다. 다만 코드를 직접 쓰지 않는다.** 구현 단계의 상세 설명은 불필요 |
| 선호 커뮤니케이션 | **결론 먼저. 분석적 말투. 표·목록 적극 활용. 이모티콘·아부 표현 금지** |
| 의사결정 패턴 | AI가 **근거와 선택지를 먼저 제시** → 마일로가 검토 후 **직접 결정** |
| 토큰 효율 | 매우 중시. 장황한 설명은 역효과 |
| 도메인 지식 | 산업안전기사 **필기 합격, 실기 준비 중** → 콘텐츠 직접 검수 가능 |
| 병행 학업 | 공조냉동기계산업기사 과정평가형 과정 (주중 전일제) — 시간 자원의 주요 제약 |

**AI가 자주 저지르는 실수 (실제 사례)**
- Claude가 "AdSense로는 강제 광고 시청 게이트를 만들 수 없다"고 단정했다가, 마일로가 제시한 스크린샷 증거로 **정정**한 이력이 있다. → **광고 정책·법령·최신 수치는 단정하지 말고 확인 후 답할 것.**
- Claude가 마일로의 학업 종료일(1월 15일)을 **서비스 출시 데드라인으로 오독**한 적이 있다. → **일정은 반드시 확인하고 가정하지 말 것.**

### 9.6 이 프로젝트를 관통하는 한 문장

> **"1인이 운영 가능한 최소 구조로, 기출 데이터가 스스로 말하게 만든다."**

모든 제약(자연 분량 원칙, 자동 렌더 섹션, 스텁 md, `/admin/` 자동 장부, 이중 원본 금지)은 결국 **혼자서 254개 챕터를 유지보수할 수 있는가**라는 하나의 질문에서 파생됐다. 새 기능을 제안할 때 이 질문을 통과하지 못하면 그 기능은 폐기 대상이다.
